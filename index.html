It's the physical scaffolding of the FMG standard. It's not just some docs to ignore; it's a directory ready to be hijacked that contains the Mandatory Minimum Anatomy. No more "I forgot the structure" excuses.

The template lives in the /template directory of this repo. Go ahead, copy it. We won't judge.

5.3 Quick Start Guide SOP ▼
  1. Clone this repo or just download the /template folder.
  2. Copy the contents of /template to your new project's root. Try not to break anything yet.
  3. Run git init. Welcome to the machine.
  4. Customize README.md and docs/SOUL.md. Give it some personality.
  5. Start your first commit as feat: initial repository setup from FMG template. Professionalism starts here.
01 · Repository Structure
1.1 Mandatory Minimum Anatomy LAW ▼

Every FMG repository exists from day one with this structure. It’s not a suggestion. It’s the law.

repo-root/
├── README.md          ← Project's public contract
├── CHANGELOG.md       ← Versioned history of changes
├── LICENSE            ← Declared license type
├── .gitignore         ← Hardened from day one
├── VERSION            ← Single source of truth for the version
├── docs/
│   └── wiki/
│       ├── index.md   ← Wiki Home
│       └── *.md       ← Domain guides
└── src/               ← Source code
Law of Completeness

A repo without a README, CHANGELOG, or .gitignore isn't a repo. It’s a folder full of regret and technical debt.

1.3 Wiki Structure (docs/wiki/) SOP ▼

The wiki documents the why and the how. The README documents the what. Don't mix them up. It's confusing.

docs/wiki/
├── index.md           ← TOC + Overview
├── architecture.md    ← Design decisions, ADRs
├── development.md     ← Local setup, contribution flow
├── agent-sop.md       ← SOP for AI agents (MANDATORY)
├── hygiene.md         ← Git hygiene, branching, releases
└── [domain].md        ← One file per domain
Law of Cohesion

Each wiki file has a single domain of responsibility. If you can't describe the file in 5 words, it's doing too much. Stop it.

Tone — Absolute Technical Authority

✓ "The system requires cargo check before commit" ✗ "It is recommended to run cargo check"
✓ "PoshBuddy guarantees clean compilation" ✗ "PoshBuddy tries to maintain clean compilation"
1.4 Hardening the .gitignore LAW ▼

Configure this in the initial commit. It's not a document to be edited only when "there's a problem." Stay ahead of the trash.

Always exclude
*.log · *.tmp · *.bak
.env* · *.local
cargo_*.txt · debug_*.txt
.manifests/ · node_modules/
__pycache__/ · target/
dist/ · build/ · .DS_Store
Never track
Credentials · API keys
Personal session files
Local build outputs
Symlinks to absolute paths
Personal IDE configs

Before every commit: run a full git status. If something unexpected shows up, add it to .gitignore before you continue. Don't be that person.

02 · Git: Commits, Versions, and Hygiene
2.1 Anatomy of an Atomic Commit LAW ▼
Golden Rule

One feature = one commit. One bugfix = one commit. One refactor = a separate commit. Docs = a separate commit. Version bump = a separate commit. Don't be lazy.

Conventional Commits Format

<type>(<scope>): <subject>

<body — WHAT and WHY, not HOW. 72 chars/line.>

<footer — Fixes #123 / BREAKING CHANGE: ...>
TypeDomainSemVer
featNew functionalityMINOR
fixBug correctionPATCH
perfPerformance optimizationPATCH
refactorReorganization without functional changeNONE
docsDocumentation, wiki, READMENONE
styleUI/visual without logic changeNONE
testNew or repaired testsNONE
choreDeps, config, release bumpsNONE
ciCI/CD pipeline changesNONE
BREAKING CHANGEIncompatible APIMAJOR

Subject: imperative, lowercase, ≤50 chars, no trailing period. Scope in lowercase: (cli), (parser), (auth). Get it right.

2.2 Semantic Versioning (SemVer) LAW ▼
ComponentWhenExample
MAJORBreaking changes. API incompatible.1.2.3 → 2.0.0
MINORNew compatible functionality.1.2.3 → 1.3.0
PATCHBugfixes. Security patches. Perf.1.2.3 → 1.2.4
Law of Single Source

The version in the project manifest (Cargo.toml, pyproject.toml, package.json, VERSION) MUST exactly match the last Git tag and the top entry in the CHANGELOG. Three sources, one number. No math required.

Law of Greatest Impact

If there's a feat and a fix in the same batch, the bump is MINOR. The rule of greatest impact always wins. Just like in real life.

Version Bump Flow

# 1. Review commits since last tag
git log --oneline vX.Y.Z..HEAD

# 2. Update version file
echo "1.3.0" > VERSION

# 3. Version commit
git add VERSION
git commit -m "chore(release): bump version to 1.3.0"

# 4. Update CHANGELOG and commit it
git add CHANGELOG.md
git commit -m "docs(changelog): update for v1.3.0"

# 5. Annotated tag
git tag -a v1.3.0 -m "Release v1.3.0"

# 6. Full push
git push origin main
git push origin v1.3.0

2.4 Branch Hygiene LAW ▼
Permanent Branches
main — stable production
dev — integration (if applicable)
Only explicitly approved branches. No "test-123" forever.
Temporary Branches
feat/short-name
fix/bug-description
refactor/component
Delete immediately post-merge. Keep it clean.

Post-Merge Cleanup Protocol

git branch -d feat/my-feature
git push origin --delete feat/my-feature
git remote prune origin
git branch -a
Anti-Force

Prohibited: git push --force to main. Ever. Only exception: isolated feature branch with extreme justification. And even then, we'll talk.

Anti-Merge Commit

Linear history is mandatory on main. Always use git pull --rebase. In shared branches: git merge. Never git pull without --rebase. Stay in line.

2.5 Full Pre-Push Protocol SOP ▼ 
# 1. Clean working tree
git status

# 2. Review commits to be pushed
git log --oneline origin/main..HEAD

# 3. Atomic? Correct messages?
git rebase -i origin/main   # If not

# 4. Successful build
cargo check / python -m py_compile / npm run build

# 5. Tests
cargo test / pytest / npm test

# 6. Second status check
git status

# 7. Push
git push origin main
git push origin vX.Y.Z

3.3 Architecture Decision Records (ADR) SOP ▼ 
# ADR 0001: Use WaveTerm as Unified GUI

**Status**: Accepted
**Date**: 2026-05-10

## Context
We need a GUI for ducer-cli without building one from scratch.

## Decision
Integrate WaveTerm as a graphical shell over the existing CLI.

## Consequences
- Positive: reduced time-to-ship, existing plugins.
- Negative: external dependency, extension limitations.
Law of Preserved Context

ADRs are never deleted or overwritten. Mark them as "Superseded by ADR XXXX". The history of decisions is primary documentation. Own your mistakes.

3.4 Docstrings and Code Comments LAW ▼
Law of Preservation

Prohibited: deleting existing comments or docstrings unless they are technically wrong. Preserving context is a higher priority than your aesthetic OCD.

Comment This
The WHY of a non-obvious decision · Workarounds with explanation · Counter-intuitive behaviors · TODOs with date and context
Don't Comment This
What the code clearly says · Obvious variable names · Every single line · Change history (that's what Git is for)
04 · AI Agent Protocol
4.1 Integration Philosophy LAW ▼
Foundational Principle

"Logic dictates. AI executes." Humans define the architecture, contracts, and decisions. The agent implements within those limits. Never the other way around. Don't let the machine drive.

The AI agent is a high-speed executor, not an architect. An agent without an SOP is just noise. An agent with an SOP is a force multiplier.

Critical Anti-Pattern

Never give an agent unrestricted write access to main without a validation pipeline. Never use an agent to make architectural decisions without human review. It's not a shortcut, it's a liability.

05 · CC Standard: Conscious Code Manifesto
5.0 Declaration of Principles CC ▼
Conscious Code Standard

Conscious code isn't just code that works. It's code that can be read, audited, maintained, and transferred without losing its intent. Human intelligence defines the intent. The tool executes the form. Don't let the tools run the asylum.

The eight principles of the CC Standard apply to all code produced under FMG, whether by human or agent.

5.1 I — Cognitive Sovereignty CC ▼
Principle I

The developer maintains total understanding of every line they sign. AI-generated code that isn't understood is never committed. No cargo culting allowed.

Using AI to generate code you don't understand is cognitive debt. Sooner or later, you'll pay for it with a bug you can't diagnose. The agent accelerates human decision-making — it doesn't replace it.

5.2 II — Failures are Primary Documentation CC ▼
Principle II

Errors are documented where they occur, with enough context to reproduce them. A bug without a post-mortem is a debt you pay twice. We hate paying twice.

# BUG: Description of what was failing and how it manifested.
# FIX: Description of the fix and why it works.
# LESSON: What was learned / what to avoid in the future.
5.3 III — Atomicity of Responsibility CC ▼
Principle III

Each function does one thing. Each module has one domain. Each commit encapsulates one change. Atomicity isn't a style — it's the minimum condition for maintainability.

Violation of atomicity is the source of most production bugs. A function that does two things has two reasons to fail. Simple math, really.

5.4 IV — Total Traceability CC ▼
Principle IV

Every non-trivial technical decision leaves a trail: in the commit, in the ADR, or in an inline comment. If a decision isn't documented, technically it never happened. Ghost code is banned.

5.5 V — Interface Contract CC ▼
Principle V

Public interfaces (APIs, CLIs, exported modules) are contracts. A breaking change without a MAJOR bump is a semantic lie. Don't lie to your users.

SemVer isn't bureaucracy. It's the mechanism by which other systems trust yours. Breaking it breaks that trust. And trust is hard to rebuild.

5.6 VI — Minimum Risk Surface CC ▼
Principle VI

The safest code is the code that doesn't exist. Every dependency is an attack surface. Every non-essential feature is maintenance debt. Simplicity is a technical decision, not an aesthetic one.

5.7 VII — Deterministic Reproducibility CC ▼
Principle VII

The same input produces the same output in any environment. Code that works "only on my machine" doesn't work. Period. Dependencies are pinned. Builds are reproducible.

Lock files (Cargo.lock, poetry.lock, package-lock.json) are always committed in final projects. No exceptions. We're not savages.

5.8

Nombres que revelan intención. Funciones que caben en una pantalla. Abstracciones que reducen carga cognitiva. Si necesitas un comentario para explicar qué hace el código (no por qué), el código falla en legibilidad.

06 · Operational Checklists
6.1 New Repository Creation SOP ▼
  • README.md created with full structure (tagline, install, usage)
  • CHANGELOG.md initialized with [Unreleased] entry
  • LICENSE declared and file present. Don't be a pirate.
  • .gitignore hardened for the project stack. No leaks allowed.
  • VERSION initialized at 0.1.0
  • Initial tag v0.1.0 created and pushed
  • Description and Topics configured on GitHub (min 3)
  • Branch main as default. Branch protection active for teams.
  • docs/wiki/index.md created if the project needs heavy documentation
  • docs/AGENT.md created if AI agents are joining the party
6.2 Pre-Commit SOP ▼
  • git status reviewed — nothing unexpected in staging
  • The change does ONE single logical thing. Focus.
  • Code compiles/runs without errors. Basic requirement, really.
  • Commit message follows Conventional Commits. Be professional.
  • Subject ≤50 chars, imperative, lowercase, no period.
  • No comments or docstrings deleted without technical reason. Preservation!
  • All AI-generated code was read and understood. No magic allowed.
6.3 Pre-Push / Release SOP ▼
  • git log --oneline origin/main..HEAD — atomic and well-named commits
  • No merge commits in history. Keep it linear.
  • VERSION updated to the new number
  • CHANGELOG.md updated with all changes
  • Project manifest version matches VERSION
  • Tag vX.Y.Z created with git tag -a
  • Tests pass (if applicable). Don't break the build.
  • git push origin main && git push origin vX.Y.Z executed
  • Post-push: git log --oneline origin/main..HEAD is empty. Job done.
6.4 Branch Hygiene (Post-Session) SOP ▼
  • All merged branches deleted locally
  • All merged branches deleted remotely
  • git remote prune origin executed. Cleanup.
  • git branch -a shows only active permanent branches
  • .gitignore updated for any new detected artifacts
6.5 Session Validation with AI Agent SOP ▼
  • Agent read AGENT.md / GEMINI.md before starting. Manners first.
  • Every operation was in minimum scope (one component at a time)
  • Agent verified compilation before reporting success. No assuming.
  • No git push --force to main from the agent. Safeguards!
  • Human reviewed and approved each architectural decision
  • All agent-generated code was read and understood by the human. Trust but verify.
6.6 Documentation Validation SOP ▼
  • Every H1 is unique per file and ≤60 chars
  • Every H2 ≤40 chars
  • No consecutive alerts in any section. Don't over-alert.
  • No absolute file:/// links or extensionless links
  • All Mermaid diagrams use PascalCase for nodes
  • Tone avoids wishy-washy verbs (could, would, might)
  • Version in CHANGELOG matches VERSION and Git tag. Triple check.

05 · FMG Repository Template
5.1 ¿Qué es el Template? REF ▼

Es el andamiaje físico del estándar FMG. No es solo documentación; es un directorio listo para ser copiado que contiene la Anatomía Mínima Obligatoria.

Ubicación

El template se encuentra en el directorio /template de este repositorio.

5.3 Guía de Inicio Rápido SOP ▼
  1. Clona este repositorio o descarga la carpeta /template.
  2. Copia el contenido de /template a la raíz de tu nuevo proyecto.
  3. Ejecuta git init.
  4. Personaliza el README.md y docs/SOUL.md.
  5. Inicia tu primer commit como feat: initial repository setup from FMG template.
1.1 Anatomía Mínima Obligatoria LEY ▼

Todo repositorio FMG existe desde el día uno con esta estructura. No es opcional.

repo-root/
├── README.md          ← Contrato público del proyecto
├── CHANGELOG.md       ← Historial versionado de cambios
├── LICENSE            ← Tipo de licencia declarado
├── .gitignore         ← Hardened desde el día uno
├── VERSION            ← Fuente única de verdad de la versión
├── docs/
│   └── wiki/
│       ├── index.md   ← Home de la wiki
│       └── *.md       ← Guías por dominio

Ley de Completitud

Un repo sin README, CHANGELOG o .gitignore configurado no es un repo. Es un directorio con deuda.

02 · Git: Commits, Versiones e Higiene
2.1 Anatomía de un Commit Atómico LEY ▼
Regla de Oro

Una feature = un commit. Un bugfix = un commit. Un refactor = un commit separado. Docs = un commit separado. Version bump = un commit separado.

Formato Conventional Commits

<type>(<scope>): <subject>

<body — QUÉ y POR QUÉ, no CÓMO. 72 chars/línea.>

<footer — Fixes #123 / BREAKING CHANGE: ...>
TipoDominioSemVer
featNueva funcionalidadMINOR
fixCorrección de bugPATCH
perfOptimización de rendimientoPATCH
refactorReorganización sin cambio funcionalNONE
docsDocumentación, wiki, READMENONE
styleUI/visual sin cambio de lógicaNONE
testTests nuevos o reparadosNONE
choreDeps, config, release bumpsNONE
ciCI/CD pipeline changesNONE
BREAKING CHANGEAPI incompatibleMAJOR

Subject: imperativo, minúscula, ≤50 chars, sin punto final. Scope en minúscula: (cli), (parser), (auth).

2.2 Versionado Semántico (SemVer) LEY ▼
ComponenteCuándoEjemplo
MAJORBreaking changes. API incompatible.1.2.3 → 2.0.0
MINORNueva funcionalidad compatible.1.2.3 → 1.3.0
PATCHBugfixes. Security patches. Perf.1.2.3 → 1.2.4
Ley de Unicidad de Fuente

La versión en el manifest del proyecto (Cargo.toml, pyproject.toml, package.json, VERSION) DEBE coincidir exactamente con el último tag de Git y la entrada superior del CHANGELOG. Tres fuentes, un número.

Ley de Mayor Impacto

Si hay un feat y un fix en el mismo lote, el incremento es MINOR. La regla de mayor impacto prevalece siempre.

Flujo de Version Bump

# 1. Revisar commits desde último tag
git log --oneline vX.Y.Z..HEAD

# 2. Actualizar archivo de versión
echo "1.3.0" > VERSION

# 3. Commit de versión
git add VERSION
git commit -m "chore(release): bump version to 1.3.0"

# 4. Actualizar CHANGELOG y commitearlo
git add CHANGELOG.md
git commit -m "docs(changelog): update for v1.3.0"

# 5. Tag anotado
git tag -a v1.3.0 -m "Release v1.3.0"

# 6. Push completo
git push origin main
git push origin v1.3.0

2.4 Higiene de Ramas LEY ▼
Ramas permanentes
main — producción estable
dev — integración (si aplica)
Solo ramas explícitamente aprobadas
Ramas temporales
feat/nombre-corto
fix/descripcion-bug
refactor/componente
Eliminar inmediato post-merge

Protocolo de Limpieza Post-Merge

git branch -d feat/mi-feature
git push origin --delete feat/mi-feature
git remote prune origin
git branch -a
Anti-Force

Prohibido git push --force a main. Único caso permitido: rama de feature aislada, bajo justificación extrema documentada.

Anti-Merge Commit

Historial lineal obligatorio en main. Usar git pull --rebase siempre. En ramas compartidas: git merge. Nunca git pull sin --rebase.

2.5 Protocolo Pre-Push Completo SOP ▼ 
# 1. Working tree limpio
git status

# 2. Revisar commits a subir
git log --oneline origin/main..HEAD

# 3. ¿Atómicos? ¿Mensajes correctos?
git rebase -i origin/main   # Si no

# 4. Compilación exitosa
cargo check / python -m py_compile / npm run build

# 5. Tests
cargo test / pytest / npm test

# 6. Segunda revisión de status
git status

# 7. Push
git push origin main
git push origin vX.Y.Z

3.3 Architecture Decision Records (ADR) SOP ▼ 
# ADR 0001: Usar WaveTerm como GUI Unificado

**Estado**: Aceptado
**Fecha**: 2026-05-10

## Contexto
Necesitamos GUI para ducer-cli sin construir una desde cero.

## Decisión
Integrar WaveTerm como shell gráfico sobre el CLI existente.

## Consecuencias
- Positivas: time-to-ship reducido, plugins existentes.
- Negativas: dependencia externa, limitaciones de extensión.
Ley del Contexto Preservado

Los ADRs nunca se eliminan ni sobreescriben. Se marcan como "Supersedido por ADR XXXX". La historia de decisiones es documentación primaria.

3.4 Docstrings y Comentarios en Código LEY ▼
Ley de Preservación

Prohibido eliminar comentarios o docstrings existentes a menos que sean técnicamente incorrectos. La preservación del contexto es prioridad sobre la limpieza estética.

Comentar
El POR QUÉ de una decisión no obvia · Workarounds con explicación · Comportamientos contra-intuitivos · TODOs con fecha y contexto
No comentar
Lo que el código claramente dice · Nombres de variables obvias · Cada línea · El historial de cambios (para eso está git)
04 · Protocolo de Agentes IA
4.1 Filosofía de Integración LEY ▼
Principio Fundacional

"Logic dictates. AI executes." El humano define la arquitectura, los contratos y las decisiones. El agente implementa dentro de esos límites. Nunca al revés.

El agente IA es un ejecutor de alta velocidad, no un arquitecto. Un agente sin SOP es ruido. Un agente con SOP es multiplicador de fuerza.

Anti-Patrón Crítico

Nunca dar a un agente acceso de escritura irrestricto a main sin pipeline de validación. Nunca usar un agente para tomar decisiones de arquitectura sin revisión humana.

05 · CC Standard: Conscious Code Manifesto
5.0 Declaración de Principios CC ▼
Conscious Code Standard

El código consciente no es aquel que funciona. Es aquel que puede ser leído, auditado, mantenido y transferido sin pérdida de intención. La inteligencia humana define la intención. La herramienta ejecuta la forma.

Los ocho principios del CC Standard aplican a todo código producido bajo FMG, ya sea por humano o agente.

5.1 I — Soberanía Cognitiva CC ▼
Principio I

El desarrollador mantiene comprensión total de cada línea que firma. El código generado por IA que no se entiende no se commitea.

Usar IA para generar código que no entiendes es deuda cognitiva. Tarde o temprano la pagas con un bug que no puedes diagnosticar. El agente acelera la ejecución de decisiones humanas — no las reemplaza.

5.2 II — Los Fallos son Documentación Primaria CC ▼
Principio II

Los errores se documentan donde ocurren, con contexto suficiente para reproducirlos. Un bug sin post-mortem es una deuda que se paga dos veces.

# BUG: descripción de qué fallaba y cómo se manifestaba.
# FIX: descripción de la corrección y por qué funciona.
# LESSON: qué se aprendió / qué evitar en el futuro.
5.3 III — Atomicidad de Responsabilidad CC ▼
Principio III

Cada función hace una cosa. Cada módulo tiene un dominio. Cada commit encapsula un cambio. La atomicidad no es un estilo — es la condición mínima de mantenibilidad.

La violación de atomicidad es el origen de la mayoría de los bugs de producción. Una función que hace dos cosas tiene dos razones para fallar.

5.4 IV — Trazabilidad Total CC ▼
Principio IV

Toda decisión técnica no trivial tiene un rastro: en el commit, en el ADR, o en un comentario inline. Si la decisión no está documentada, técnicamente no existió.

5.5 V — Contrato de Interfaces CC ▼
Principio V

Las interfaces públicas (APIs, CLIs, módulos exportados) son contratos. Un cambio breaking sin MAJOR bump es una mentira semántica.

SemVer no es burocracia. Es el mecanismo por el cual otros sistemas confían en el tuyo. Romperlo es romper esa confianza.

5.6 VI — Mínima Superficie de Riesgo CC ▼
Principio VI

El código más seguro es el que no existe. Cada dependencia es una superficie de ataque. Cada feature no esencial es deuda de mantenimiento. La simplicidad es una decisión técnica, no estética.

5.7 VII — Reproducibilidad Determinista CC ▼
Principio VII

El mismo input produce el mismo output en cualquier ambiente. El código que funciona "solo en mi máquina" no funciona. Las dependencias se pinean. Los builds son reproducibles.

Lock files (Cargo.lock, poetry.lock, package-lock.json) se commitean siempre en proyectos finales. Sin excepción.

5.8 VIII — Legibilidad como Feature CC ▼
Principio VIII

El código se escribe una vez y se lee cien veces. La legibilidad no es un lujo posterior — es parte del diseño inicial. El código inteligible sobrevive a su autor.

Nombres que revelan intención. Funciones que caben en una pantalla. Abstracciones que reducen carga cognitiva. Si necesitas un comentario para explicar qué hace el código (no por qué), el código falla en legibilidad.

06 · Checklists Operativos
6.1 Creación de Nuevo Repositorio SOP ▼
  • README.md creado con estructura completa (tagline, install, uso)
  • CHANGELOG.md inicializado con entrada [Unreleased]
  • LICENSE declarado y archivo presente
  • .gitignore hardened para el stack del proyecto
  • VERSION inicializado en 0.1.0
  • Tag inicial v0.1.0 creado y pusheado
  • Descripción y Topics configurados en GitHub (mínimo 3)
  • Branch main como default. Branch protection activa si es colaborativo.
  • docs/wiki/index.md creado si el proyecto requiere wiki extensa
  • docs/AGENT.md creado si el proyecto integra agentes IA
6.2 Pre-Commit SOP ▼
  • git status revisado — nada inesperado en staging
  • El cambio hace UNA sola cosa lógica
  • El código compila/ejecuta sin errores
  • Mensaje de commit sigue Conventional Commits
  • Subject ≤50 chars, imperativo, minúscula, sin punto final
  • No se eliminaron comentarios o docstrings sin razón técnica
  • Todo código generado por IA fue leído y comprendido
6.3 Pre-Push / Release SOP ▼
  • git log --oneline origin/main..HEAD — commits atómicos y bien nombrados
  • No hay merge commits en el historial
  • VERSION actualizado al nuevo número
  • CHANGELOG.md actualizado con todos los cambios
  • Versión en manifest del proyecto coincide con VERSION
  • Tag vX.Y.Z creado con git tag -a
  • Tests pasan (si aplica)
  • git push origin main && git push origin vX.Y.Z ejecutados
  • Post-push: git log --oneline origin/main..HEAD está vacío
6.4 Higiene de Ramas (Post-Sesión) SOP ▼
  • Todas las ramas mergeadas fueron eliminadas localmente
  • Todas las ramas mergeadas fueron eliminadas en remoto
  • git remote prune origin ejecutado
  • git branch -a muestra solo ramas permanentes activas
  • .gitignore actualizado para cualquier nuevo artefacto detectado
6.5 Validación de Sesión con Agente IA SOP ▼
  • El agente leyó AGENT.md / GEMINI.md antes de operar
  • Cada operación fue en scope mínimo (un componente a la vez)
  • El agente verificó compilación antes de reportar éxito
  • No se ejecutó git push --force a main desde el agente
  • El humano revisó y aprobó cada decisión de arquitectura
  • Todo código generado por agente fue leído y comprendido por el humano
6.6 Validación de Documentación SOP ▼
  • Todo H1 es único por archivo y ≤60 chars
  • Todo H2 ≤40 chars
  • No hay alertas consecutivas en ninguna sección
  • No hay links absolutos file:/// o sin extensión
  • Todos los diagramas Mermaid usan PascalCase en nodos
  • El tono no usa verbos de duda (podría, sería, intentaría)
  • La versión en CHANGELOG coincide con VERSION y el tag de Git
